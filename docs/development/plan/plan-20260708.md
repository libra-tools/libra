# Git compatibility gap closure plan (2026-07-08)

## 文档职责

本文把 `/Users/eli/Downloads/libra_git.md` 中的 Git 对比探针结果收敛成 Libra 的兼容改进计划。目标不是追求 100% 复刻 Git 的全部长尾 surface，而是优先修复会导致静默错误、历史形态漂移、CI/脚本误判、用户漏看冲突的高风险差距。

本文只规划任务，不宣称实现完成。落地时必须同步：`COMPATIBILITY.md`、`docs/development/commands/_compatibility.md`、对应 `docs/development/commands/<cmd>.md`、用户命令文档、`tests/INDEX.md`、Cargo test target 和 compat/integration 测试。

## 明确不支持范围

以下 Git surface 明确不进入本计划，不作为 P0/P1/P2 改进项：

| Git surface | 决策 | 依据 |
|---|---|---|
| `submodule` 命令族 | 明确不支持 | Libra 产品边界是单仓库 / trunk-based；维持 `docs/development/commands/_compatibility.md` D1。 |
| `clone --recurse-submodules` / `clone --recursive` | 明确不支持 | 依赖 `submodule`，维持 D4。 |
| `subtree` 命令 | 明确不支持 | 不加入兼容改进计划；如需组合能力，应走 Libra 自身的 link/subtree composition RFC，而不是 Git `subtree` CLI 兼容。 |
| `filter-branch`、`whatchanged`、`instaweb`、`citool` 等老旧或 GUI 命令 | 不纳入本计划 | 维持 unsupported，除非后续有独立用户需求和 RFC。 |
| 交互式 patch mode：`add -p`、`commit -p`、`reset -p`、`restore -p`、`stash -p` | 本计划不重启 | 维持 D15，当前优先非交互式、Agent 可驱动路径。 |
| 交互式 rebase / todo 编辑：`rebase -i`、`--edit-todo` | 本计划不重启 | 维持 D16；可脚本化 rebase 选项可在 P1 做，交互式 todo 不做。 |
| materializing sparse-checkout | 本计划不重启 | 维持 D10；`sparse-view` 是只读显示过滤，不等价于 Git sparse-checkout。 |

## 兼容等级调整任务

### CG-01 重新校准命令级兼容等级

- 目标：把 `COMPATIBILITY.md` 从粗粒度命令级承诺，调整为能表达关键子面的分级承诺，避免 `supported` 掩盖冲突、porcelain、config、plumbing 兼容缺陷。
- 范围：`COMPATIBILITY.md`、`docs/development/commands/_compatibility.md`、对应命令文档。
- 建议新增子面标签：
  - `common-user-flow`：常规人工路径是否可用。
  - `porcelain-machine`：`--porcelain`、`-z`、exit code、stdout/stderr 是否可脚本解析。
  - `conflict-aware`：unmerged index、sequencer、restore/diff/status 是否正确。
  - `config-aware`：是否遵守 Git 常用 config 默认值。
  - `plumbing-compatible`：对象、ref、rev syntax、raw 输出是否接近 Git。
- 必须下调或拆分的候选：
  - `status`：当前不能整体标为强 `supported`，因为 conflict status、porcelain v2 XY、`status.*` config、`rm --cached` 展示均有差异。
  - `diff`：默认 untracked、`--check`、raw/name-status/numstat、rename、pickaxe 等差异需要拆分。
  - `commit`：author/date env、hooks、dry-run、amend、message source 等差异需要拆分。
  - `fetch` / `pull` / `merge` / `rebase`：config 语义、refspec 语义、sequencer/conflict 状态需要拆分。
- 验收：兼容矩阵中每个被 P0/P1 触达的命令都有“已支持面 / 部分支持面 / 明确不支持面 / 有意差异面”；新增或保留的 unsupported 项必须在 `_compatibility.md` 有 D 编号或明确计划编号。

## P0：安全与静默错误修复

P0 只收录可能造成漏冲突、损坏仓库、错误历史、CI 误判或命令成功但实际未执行语义的项目。

### P0-01 修复 unmerged/conflict 状态与 diff 展示

- 问题：merge/rebase/cherry-pick 冲突路径在 `status --porcelain` 中可被报成 `??`，`diff` 把冲突文件当作从 `/dev/null` 新增，脚本和用户会漏看冲突。
- 范围：`status`、`diff`、`ls-files -u/-t`、merge/rebase/cherry-pick sequencer 状态。
- 要求：
  - `status --porcelain=v1` 对内容冲突输出 `UU`。
  - `status --porcelain=v2` 输出 Git 兼容 unmerged record。
  - `diff` 支持 unmerged stage 视图，至少不能把冲突误报为 untracked add。
  - `ls-files -t` 不得隐藏 unmerged entries。
- 测试：新增 `tests/compat/conflict_status_diff_test.rs`，覆盖 merge、rebase、cherry-pick 三条路径。

### P0-02 补齐 `diff --check` 的安全检查

- 问题：`diff --check` 漏检 leftover conflict markers 和 new blank line at EOF，导致 pre-commit/CI 误放行。
- 范围：`src/command/diff*` 共享检查逻辑。
- 要求：对齐 Git 的至少三类检查：trailing whitespace、conflict marker、blank line at EOF。
- 测试：新增 `tests/compat/diff_check_safety_test.rs`。

### P0-03 禁止产生 broken shallow clone

- 问题：local `clone --depth 1` 可创建无法遍历、`fsck` 报 missing object 的仓库。
- 范围：`clone`、fetch object closure、shallow metadata。
- 要求：
  - 如果浅克隆对象闭包不完整，命令必须失败并回滚目标目录或标记为不可用。
  - 若暂不支持某类 shallow source，必须 fail closed，不得留下 broken repo。
- 测试：新增 `tests/compat/clone_shallow_integrity_test.rs`。

### P0-04 修复 checkout 分支创建后 detached HEAD

- 问题：`checkout -b <branch> <start-point>` 和 `checkout -B <branch> <start-point>` 可成功但留下 detached HEAD。
- 范围：`checkout` 与 `switch` 分支创建共享逻辑。
- 要求：Git 兼容拼写创建/重置分支后 HEAD 必须指向该分支；失败时不得移动 HEAD。
- 测试：`tests/compat/checkout_branch_startpoint_test.rs`。

### P0-05 修复 orphan 分支语义

- 问题：`switch --orphan` 可保留旧 parent，违背 root branch 语义。
- 范围：`switch --orphan`、index/worktree 初始化、后续 commit parent。
- 要求：orphan 后首个 commit 无 parent；若无法保证，命令必须拒绝。
- 测试：`tests/compat/switch_orphan_root_test.rs`。

### P0-06 处理 stdout broken pipe，不得 panic

- 问题：`cat-file -p HEAD | head -n 1` 等正常 Unix pipeline 可触发 panic。
- 范围：全局输出层、`cat-file`、其他大输出命令。
- 要求：`EPIPE` 视为正常下游提前关闭；不得打印 panic；exit 行为对齐常见 Unix 工具。
- 测试：`tests/compat/broken_pipe_output_test.rs`。

### P0-07 修复 `commit --amend --no-edit` 的 no-op 成功问题

- 问题：clean amend 可打印成功但 HEAD 不变，脚本误以为刷新了 committer metadata/signature/hooks。
- 范围：`commit --amend`。
- 要求：Git 兼容重写 commit；若没有任何可重写内容且产品决定 no-op，必须清晰提示且不得打印真实提交摘要。
- 测试：`tests/compat/commit_amend_no_edit_test.rs`。

### P0-08 修复历史写入身份和日期保真

- 问题：`GIT_AUTHOR_*`、`GIT_COMMITTER_DATE`、`--date`、`--reset-author`、`commit -C/-c`、`cherry-pick`/`revert` 可能丢失作者、日期或原始 subject。
- 范围：commit object builder、sequencer commit creation。
- 要求：
  - 支持 Git 常用 author/date env override。
  - `--date` 设置 author date。
  - `--reset-author` 对 amend 生效。
  - `-C/-c` 按 Git 语义复用 message 和 author metadata。
  - cherry-pick/revert 生成消息不得从 PGP block 或空 subject 派生错误 subject。
- 测试：`tests/compat/commit_identity_date_test.rs`、`tests/compat/sequencer_message_author_test.rs`。

### P0-09 修复 `write-tree` / object integrity 前置校验

- 问题：index entry 指向 missing object 时 `write-tree` 可能成功。
- 范围：`write-tree`、commit 前 index object validation。
- 要求：写 tree 和 commit 前校验所有 index object 存在且类型正确。
- 测试：`tests/compat/write_tree_missing_object_test.rs`。

### P0-10 修复 `init --shared=<numeric>` partial broken repo

- 问题：`init --shared=0660` 可留下无 execute bit 的不可用 `.libra`。
- 范围：`init --shared`。
- 要求：预校验 numeric mode 对目录可遍历性；失败必须回滚 partial repo。
- 测试：`tests/compat/init_shared_mode_test.rs`。

### P0-11 修复 symlink 基础兼容

- 问题：status/add 可忽略 symlink，checkout 可能把 tracked symlink 写为普通文件。
- 范围：index mode `120000`、checkout、status、add、diff。
- 要求：至少对 Unix symlink 做 Git 兼容读写；不支持平台必须显式 skip/diagnostic。
- 测试：`tests/compat/symlink_basic_test.rs`。

## P1：高价值 Git 兼容增强

P1 关注日常脚本、CI、工具集成和团队配置，不包含明确拒绝的 submodule/subtree/interactive patch/rebase-i。

### P1-01 共享 Git pathspec engine

- 目标：统一 `add`、`rm`、`status`、`diff`、`grep`、`ls-files`、`checkout`、`restore`、`clean` 的 pathspec 行为。
- 覆盖：glob、`--` separator、`:(exclude)`、`:(top)`、`:(icase)`、`:(literal)`、`:(glob)`、从子目录运行时的路径相对语义。
- 非目标：不重启 `clean <pathspec>` 的删除行为前，先只用于 dry-run/只读命令或已支持 pathspec 的命令。
- 测试：`tests/compat/pathspec_magic_test.rs`。

### P1-02 兼容 Git ignore 和 attributes 来源

- 目标：补齐 `.gitignore`、`.git/info/exclude`、`core.excludesFile`、`.gitattributes`、`.git/info/attributes`、`core.attributesFile` 的读取。
- Libra 扩展：`.libraignore` 和 `.libra_attributes` 可继续存在，但不能让 Git 标准文件完全失效。
- 覆盖命令：`status`、`add`、`clean`、`check-ignore`、`check-attr`、`archive`、`diff --textconv`、LFS/media 相关属性读取。
- 测试：`tests/compat/ignore_attributes_sources_test.rs`。

### P1-03 对齐 machine porcelain 输出

- 目标：让脚本依赖的输出稳定 Git-like。
- 覆盖：
  - `status --porcelain=v1/v2 -z`。
  - `diff --quiet` / `--exit-code` 不因 untracked `.libraignore` 假失败。
  - `diff --name-status` / `--numstat` 默认不包含 untracked。
  - `ls-files --error-unmatch` exit code。
  - `grep` no-match exit code。
- 测试：`tests/compat/machine_porcelain_contract_test.rs`。

### P1-04 补齐 pretty-format renderer

- 目标：`log` / `show` / `shortlog` / `for-each-ref` 不再把常用 Git placeholders 原样打印。
- 优先 placeholders：`%b`、`%B`、`%n`、`%xNN`、`%aI`、`%cI`、`%at`、`%ct`、`%p`、`%P`、`%D`、`%m`、color placeholders。
- 要求：未支持 placeholder 要么按 Git literal 规则有测试说明，要么明确报 unsupported；不能误导机器输出。
- 测试：`tests/compat/pretty_format_placeholders_test.rs`。

### P1-05 遵守高影响 Git config 默认值

- 目标：消除“用户设置了 Git config 但 Libra 忽略，导致历史或输出改变”的风险。
- 第一批 config：
  - `init.defaultBranch`。
  - `pull.rebase`、`branch.<name>.rebase`、`pull.ff`。
  - `merge.ff`、`merge.log`、`merge.verifySignatures`。
  - `fetch.prune`、`remote.<name>.prune`。
  - `status.showUntrackedFiles`、`status.showStash`、`status.branch`、`status.short`、`status.relativePaths`。
  - `branch.sort`、`tag.sort`。
  - `diff.renames`、`diff.context`、`diff.noprefix`、`diff.mnemonicPrefix`、`diff.srcPrefix`、`diff.dstPrefix`。
  - `format.pretty`、`log.date`、`log.follow`。
  - `commit.status`、`commit.cleanup`、`commit.gpgSign`。
- 测试：`tests/compat/config_defaults_semantics_test.rs`。

### P1-06 修复 fetch / remote refspec 精确性

- 目标：fetch 行为按 Git refspec 和 remote config 精确更新。
- 覆盖：
  - `fetch <remote> <src>:<dst>`。
  - `remote add -t` / `remote set-branches` 后 fetch 只更新配置分支。
  - `remote update` 遵守 `remotes.default`。
  - `remote rename` 迁移 remote-tracking refs。
  - `origin/HEAD` / `ls-remote --symref` / `FETCH_HEAD` / `ORIG_HEAD` 关键元数据。
- 测试：`tests/compat/fetch_remote_refspec_test.rs`。

### P1-07 补齐 non-interactive history controls

- 目标：只补可脚本化历史控制，不做 interactive rebase/todo。
- 覆盖：
  - `rebase --autostash`、`--exec`、`--update-refs`、`--fork-point`。
  - `merge -X ours/theirs`、`merge -s ours`、`--allow-unrelated-histories`、`--log`。
  - `cherry-pick -X`、`revert -X`、`revert --cleanup`。
  - `reset --merge`、`reset --keep`。
- 测试：`tests/compat/noninteractive_history_controls_test.rs`。

### P1-08 补齐 common diff/review options

- 目标：支持代码审查和 CI 常用 diff surface。
- 覆盖：`--raw`、`--summary`、`--compact-summary`、`--diff-filter`、`-S`、`-G`、`--color-words`、`--word-diff-regex`、`--src-prefix`、`--dst-prefix`、`--full-index`、`--minimal`、`--patience`、`--histogram`、`--anchored`。
- 测试：`tests/compat/diff_review_options_test.rs`。

### P1-09 修复 tag / rev / object peel 语义

- 目标：annotated tag、peel syntax、rev selectors 对脚本可用。
- 覆盖：`rev-parse HEAD^{commit/tree}`、`REV:path`、`@` shorthand、reflog selectors、tag refs、`cat-file` peel syntax、`branch --points-at <annotated-tag>`、`show-ref --dereference`。
- 测试：`tests/compat/rev_parse_peel_selectors_test.rs`。

### P1-10 Hooks 兼容层决策与最小实现

- 目标：当前 `_compatibility.md` D3 表示 Git hooks bridge 不作为核心默认能力；但 `libra_git.md` 显示 hooks 被忽略会导致 policy 失效。需要明确产品决策并实现最小安全路径。
- 选项 A：保持不读 `.git/hooks`，但完整支持 `.libra/hooks` 中同名生命周期：`prepare-commit-msg`、`post-commit`、`post-checkout`、`pre-rebase`、`pre-merge-commit`、`post-merge`、`post-rewrite`。
- 选项 B：增加 opt-in Git hooks bridge，例如 `hooks.gitCompatibility=true`，默认关闭。
- 本计划建议：先做 A，并在 `COMPATIBILITY.md` 明确 `.git/hooks` 非默认；如用户需要再评估 B。
- 测试：`tests/compat/libra_hooks_lifecycle_test.rs`。

### P1-11 Import/export 基础保真

- 目标：迁移和备份常用路径不丢 tag、notes、路径 quoting、range 语义。
- 覆盖：
  - `fast-export` 支持 ranges、多个 revisions、tag objects、notes refs、path quoting、incremental stream shape。
  - `fast-import` 支持 quoted paths、inline blob data、rename/copy commands、tag commands、notes modify。
  - `bundle create --all/--branches/--tags`、`bundle unbundle`。
- 测试：`tests/compat/import_export_roundtrip_test.rs`。

## P2：长尾命令，仅邮件系列

P2 只加入邮件补丁流命令。除本节外，其他长尾 Git 命令维持 unsupported 或 intentionally-different，不进入本计划。

### P2-01 `am` 最小实现

- 目标：支持应用 `format-patch` 生成的 email patch，覆盖最常用邮件补丁流。
- 范围：`libra am <patch...>`、`--abort`、`--continue`、`--skip` 的最小 sequencer；先不支持全部 Git mail flags。
- 依赖：P2-02 `mailinfo` 或内部 mail parser。
- 测试：`tests/compat/mail_am_basic_test.rs`。

### P2-02 `mailinfo` 最小实现

- 目标：解析 email patch 的 commit message 与 patch body，供 `am` 和脚本使用。
- 范围：`mailinfo msg patch < mail`，基本 headers、subject cleanup、body split、patch extraction。
- 测试：`tests/compat/mailinfo_basic_test.rs`。

### P2-03 `format-patch` 邮件输出补齐

- 目标：让 Libra 生成的 patch 能被 Git 和 Libra `am` 可靠消费。
- 覆盖：`format-patch -1 --stdout` shorthand、cover letter 基础、threading headers、`--in-reply-to`、`format.subjectPrefix`、`format.signOff`、`format.outputDirectory`、`format.suffix`、MIME boundary 的基础兼容。
- 测试：`tests/compat/format_patch_mail_roundtrip_test.rs`。

### P2-04 `send-email` 明确策略

- 目标：决定是否实现真正发送，还是只提供 dry-run / file output 辅助。
- 建议：先不做 SMTP 发送；实现 `send-email --dry-run` / `--validate-only` 可选，或明确继续 unsupported。
- 验收：必须在 `COMPATIBILITY.md` 说明状态，避免用户误以为完整支持。

## 实施顺序

1. CG-01 先行：调整兼容等级表达，避免继续扩大错误承诺。
2. P0-01 到 P0-04：先修冲突、diff check、shallow clone、checkout detached 这四个最高风险项。
3. P0-05 到 P0-11：修历史写入、安全输出、object integrity、init shared、symlink。
4. P1-01 到 P1-03：统一 pathspec/ignore/porcelain，这是多命令差异的共同底座。
5. P1-04 到 P1-06：修 pretty format、config defaults、fetch/remote。
6. P1-07 到 P1-11：修非交互历史控制、diff review、rev peel、hooks、import/export。
7. P2 邮件系列：在 P0/P1 收敛后启动；不夹带 `submodule`、`subtree` 或其他长尾命令。

## 验收门禁

- 每个任务必须有对应 compat test，不接受只更新文档。
- 每个改动必须同步 `COMPATIBILITY.md` 和对应命令文档。
- 涉及用户可见错误时，必须使用稳定 `LBR-*` 错误码并更新 `docs/error-codes.md`。
- 必跑：`cargo +nightly fmt --all --check`、`cargo clippy --all-targets --all-features -- -D warnings`、`cargo test --all`。
- 新增/改名 test target 时同步 `Cargo.toml` 和 `tests/INDEX.md`。

## 非目标提醒

- 不以 `libra_git.md` 的 1100+ 项逐项清零作为成功标准。
- 不把 Git 老旧命令和辅助 helper 全量搬进 Libra。
- 不用“有意差异”掩盖静默错误：凡是命令成功但状态、历史、对象或机器输出误导用户，都必须修或降级承诺。
